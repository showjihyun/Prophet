# 에이전트 엔진 및 확산 엔진 비교 분석

> CAMEL-OASIS (MiroFish) vs Prophet 자체 엔진 기술 비교 및 선택 가이드

## 1. 에이전트 엔진 비교

| 비교 항목 | CAMEL-OASIS | Prophet 자체 6-Layer |
|-----------|-------------|---------------------|
| **에이전트 구조** | LLM 단일 추론 (CoT) | 6-Layer 파이프라인 (Perception->Memory->Emotion->Cognition->Decision->Influence) |
| **의사결정 방식** | 매 턴 LLM 호출 (Chain-of-Thought) | 3-Tier (80% Rule / 10% Heuristic / 10% LLM) |
| **액션 공간** | 21종 (sign_up, repost, follow, mute 등 SNS 특화) | 6종 계층 (ignore < view < like < comment < share < adopt) |
| **감정 모델** | 없음 (LLM에 암묵적 위임) | 명시적 4차원 (interest, trust, skepticism, excitement) |
| **메모리** | CAMEL ScoreBasedContextCreator + Zep Cloud | GraphRAG + pgvector 임베딩 (자체 관리) |
| **성격 모델** | 프로필 텍스트 기반 (LLM이 해석) | 5차원 벡터 (openness, skepticism, trend_following, brand_loyalty, social_influence) |
| **시간 모델** | 24차원 활동 벡터 (시간대별 활성 확률) | Step 기반 (시간대 개념 없음) |
| **확장성** | 100만 에이전트 (27x A100 필요) | 1,000 에이전트 목표 (단일 서버) |
| **추론 엔진** | vLLM 분산 추론 | Ollama / Claude / OpenAI API |
| **실행 비용** | 모든 에이전트에 LLM 호출 -> 매우 높음 | 10%만 LLM -> 90% 절감 |
| **해석 가능성** | CoT 로그로 추론 과정 추적 | 각 Layer 출력값으로 단계별 추적 |
| **검증 상태** | 실제 Twitter15/16 데이터셋 검증 (NRMSE ~30%) | 스펙 단계 (미검증) |

## 2. 확산 엔진 비교

| 비교 항목 | OASIS 내장 | Prophet Viral Cascade Algorithm |
|-----------|-----------|--------------------------------|
| **확산 모델** | **창발적(Emergent)** — 수학 모델 없음, LLM 결정의 집합 효과 | **수학적(Mathematical)** — `P(i->j) = I_i * trust_ij * emotion_factor * msg_strength` |
| **콘텐츠 노출** | RecSys 알고리즘 (Twitter/Reddit 방식) | Exposure Model (`P = Sigma influence_j * W_ij`) |
| **전파 경로** | Follow 관계 + RecSys 추천 | 네트워크 엣지 + 커뮤니티 브리지 |
| **캐스케이드 감지** | 없음 (사후 분석 필요) | **자동 감지** (Viral, Polarization, Echo Chamber, Collapse) |
| **네트워크 생성** | 코어유저 기반 + 합성 유저 (follow 확률 0.2) | Watts-Strogatz + Barabasi-Albert 하이브리드 |
| **네거티브 확산** | LLM이 자체 판단 | 명시적 수식 `P_neg = skepticism * controversy * influencer_effect` |
| **전문가 개입** | ManualAction으로 수동 주입 | Expert Agent 타입 (Reviewer/Scientist/Journalist) + Tier3 LLM |
| **플랫폼 시뮬** | Twitter + Reddit 충실 재현 | 플랫폼 비특정 (마케팅 캠페인 범용) |
| **검증 데이터** | Twitter15/16 실데이터 비교 완료 | 미검증 |

## 3. OASIS 상세 기술 분석

### 3.1 5대 핵심 컴포넌트

| 컴포넌트 | 역할 |
|----------|------|
| **Environment Server** | SQLite 기반 시뮬레이션 상태 관리 (프로필, 포스트, 팔로우, 추천) |
| **Agent Module** | LLM 기반 에이전트 (Llama3-8B-instruct 기본), CoT 추론 |
| **Recommendation System** | Random / Twitter-style (TwHIN-BERT) / Reddit-style (Hot Score) |
| **Time Engine** | 24차원 활동 벡터, 1스텝 = 실시간 3분 |
| **Scalable Inferencer** | vLLM 분산 추론, 비동기 메시지 큐 |

### 3.2 RecSys 알고리즘 상세

- **Twitter-style**: In-network 인기도 순위 + Out-of-network TwHIN-BERT 임베딩 유사도
- **Reddit-style**: `h = log10(max(|u-d|, 1)) + sign(u-d) * (t-t0) / 45000`
- **Random**: 무작위 노출 (baseline)

### 3.3 OASIS 검증 결과

- 평가 데이터: Twitter15/Twitter16 (198개 실제 전파 사례, 9개 카테고리)
- 측정 메트릭: Scale (참여 규모), Depth (전파 깊이), Max Breadth (최대 폭)
- 결과: **NRMSE ~30%** (현실 데이터와 정렬)
- 한계: 시뮬레이션 Depth가 현실 대비 일관되게 낮음

### 3.4 OASIS 알려진 한계

1. **RecSys 충실도**: 중간 규모 유저 모델링 정밀도 부족
2. **LLM 편향**: 비검열 모델은 극단화, 정렬 모델은 과도한 중립화
3. **전파 깊이**: 현실 대비 일관되게 낮음
4. **시간 해상도**: 3분 단위 → 세밀한 실시간 행동 포착 제한
5. **플랫폼 범위**: Twitter/Reddit만 지원, 기타 플랫폼은 커스텀 RecSys 필요
6. **합성 프로필**: 실제 인간 행동 복잡성 재현 한계
7. **컴퓨팅 비용**: 100만 에이전트 = A100 27대, 1스텝에 18시간
8. **메모리**: 40+ 라운드 시 16GB RAM 권장

## 4. Prophet 자체 엔진 상세 기술 분석

### 4.1 6-Layer Agent Architecture

#### Layer 1: Perception (지각)
- 환경 이벤트 필터링 및 가중치 부여
- 입력: campaign_ad, influencer_post, expert_review, community_discussion
- 출력: `PerceptionResult` (feed_items, social_signals, expert_signals, total_exposure_score)

#### Layer 2: Memory (기억)
- GraphRAG 스타일 그래프 저장소
- 메모리 유형: Episodic (경험) / Semantic (지식) / Social (관계)
- 검색 알고리즘: `Score = alpha*recency + beta*relevance(pgvector cosine) + gamma*emotion_weight + delta*social_importance`
- Top-K 검색 (기본 K=10)

#### Layer 3: Emotion (감정)
- 4차원: interest, trust, skepticism, excitement (모두 0.0-1.0)
- 업데이트: `E_i(t+1) = E_i(t) + social_signal + media_signal + expert_signal - decay`
- 확산 감정 팩터: `emotion_factor = excitement - skepticism`

#### Layer 4: Cognition (인지) — 3-Tier LLM 실행

| Tier | 에이전트 비율 | 모델 | 비용 | 지연 | 용도 |
|------|-------------|------|------|------|------|
| Tier 1 | ~80% | Rule Engine | $0 | ~5ms | 일반 에이전트, 빠른 결정 |
| Tier 2 | ~10% | Heuristic | $0 | ~10ms | 복잡한 성격 가중치 결정 |
| Tier 3 | <=10% | LLM | $$ | 100-500ms | 인플루언서, 전문가 추론 |

- Tier 1 수식: `evaluation = interest + trust - skepticism + community_bias + memory_weight`
- Tier 3 타임아웃 시 Tier 2로 자동 폴백

#### Layer 5: Decision (결정)
- 행동 계층: ignore < view < like < comment < share < adopt
- 확률: `P(action) = softmax(evaluation_score + social_pressure)`
- 사회적 압력: `social_pressure = Sigma (influence_j * trust_ij * action_weight_j)`

#### Layer 6: Influence (영향력 전파)
- 전파 확률: `P(i->j) = influence_i * trust_ij * emotion_factor * message_strength`
- 메시지 강도: `message_strength = (novelty + controversy + utility) / 3.0`

### 4.2 Viral Cascade Algorithm

#### 확산 파이프라인
```
Content Event -> Exposure Model -> Cognitive Evaluation -> Behavior Decision -> Social Propagation -> Cascade Detection
```

#### 자동 캐스케이드 감지
1. **Viral Cascade**: `cascade_size > threshold`
2. **Polarization**: `sentiment_variance > threshold`
3. **Echo Chamber**: `community_internal_links >> external_links`
4. **Slow Adoption**: 장기간 점진적 확산
5. **Collapse**: 부정 리뷰/전문가 개입으로 급격한 채택 중단

#### 네거티브 확산
```
P_neg(i->j) = skepticism_i * controversy * influencer_effect
```

### 4.3 하이브리드 네트워크 생성

```
Global Network
   ├─ Community A-E (Watts-Strogatz small-world)
   │   - N = 100-500, K = 6, p = 0.1
   │   - 클러스터링 계수: 0.2-0.6
   │   - 평균 경로 길이: 4-6
   ├─ Influencer Layer (Barabasi-Albert scale-free)
   │   - 1% > 100K 팔로워, 9% 100-10K, 90% < 100
   ├─ Cross-Community Bridges (P_cross = 0.02)
   └─ Edge Weight: W_ij = trust_ij * interaction_freq
```

## 5. 핵심 트레이드오프 분석

### 5.1 정밀도 vs 비용

```
OASIS:  높은 정밀도 + 높은 비용
        ├─ 장점: 모든 에이전트가 LLM 추론 -> 현실적 행동 패턴
        ├─ 단점: 100K 유저, A100 5대, 10스텝에 ~2일 소요
        └─ 비용: GPU 인프라 또는 API 비용 막대

Prophet: 적절한 정밀도 + 낮은 비용
        ├─ 장점: 80% Rule Engine -> 단일 서버에서 1,000 에이전트 <= 1초
        ├─ 단점: Rule Engine 에이전트 행동이 단순할 수 있음
        └─ 비용: LLM 비용 90% 절감
```

### 5.2 유연성 vs 제어력

```
OASIS:  높은 유연성 + 낮은 제어력
        ├─ 장점: LLM이 예상치 못한 창발적 행동 생성
        ├─ 장점: 21종 액션으로 SNS 행동 세밀 재현
        └─ 단점: 확산 경로를 수학적으로 예측/제어 불가

Prophet: 적절한 유연성 + 높은 제어력
        ├─ 장점: 수학적 확산 모델로 예측 가능
        ├─ 장점: 감정 상태, 성격 벡터를 직접 조작 가능
        └─ 단점: 사전 정의된 6종 액션으로 제한됨
```

### 5.3 마케팅 적합성

```
OASIS:  SNS 플랫폼 시뮬레이션에 최적화
        ├─ Twitter/Reddit 충실 재현
        ├─ RecSys로 콘텐츠 도달 현실적 시뮬
        └─ 마케팅 메트릭 없음 (직접 구현 필요)

Prophet: 마케팅 캠페인 분석에 최적화
        ├─ 바이럴 확률, 채택률, 감성분석 내장
        ├─ Monte Carlo로 확률 분석
        ├─ 실시간 개입 (캠페인 전략 테스트)
        └─ 특정 SNS 플랫폼 재현은 약함
```

## 6. 추천: 하이브리드 전략

순수하게 하나만 선택하기보다, Prophet의 목표에 맞춘 하이브리드 접근을 추천한다.

│  [자체 유지]               [OASIS 참조/차용]      │
│                                                  │
│  ✓ 6-Layer Agent           ◐ RecSys 개념 도입    │
│    Architecture              (콘텐츠 노출 현실화) │

│  ✓ 3-Tier LLM              ◐ 24h Activity Vector │

│  ✓ 감정/성격 수학 모델      ✗ OASIS 직접 의존    │

### 6.2 선택 근거

| 판단 기준 | 결론 | 근거 |
|-----------|------|------|
| **에이전트 엔진** | **자체 6-Layer 유지** | Prophet의 감정/성격 수학 모델은 마케팅 분석에 핵심. OASIS의 "LLM에 모두 위임" 방식은 비용 과다 + 재현성 낮음 |
| **확산 엔진** | **자체 Viral Cascade 유지** | 수학적 확산 모델이 있어야 Monte Carlo 분석, 캐스케이드 자동감지, 마케팅 KPI 추출 가능. OASIS의 창발적 모델은 "왜 퍼졌는가" 설명 불가 |
| **LLM 비용** | **3-Tier 유지** | SaaS 모델에서 OASIS처럼 모든 에이전트에 LLM 호출은 사업적으로 비현실적 |
| **RecSys** | **OASIS에서 차용** | 현재 Prophet의 Exposure Model이 너무 단순. RecSys 개념을 추가하면 콘텐츠 노출 현실성 향상 |
| **시간 모델** | **OASIS에서 차용** | 24시간 활동 벡터는 구현 비용 낮고 시뮬레이션 현실성 크게 향상 |
| **액션 공간** | **확장 검토** | 6종은 마케팅에 충분하나, search, follow/unfollow 정도 추가 검토 가치 있음 |
| **OASIS 직접 의존** | **회피** | 런타임 의존은 커스터마이징 자유도와 비용 제어력을 잃음 |

### 6.3 OASIS에서 구체적으로 차용할 아이디어

#### A. RecSys 계층 추가

Prophet의 PerceptionLayer에 추천 알고리즘 로직 삽입:

```
PerceptionLayer.observe()
  └─ 기존: exposure_score = Sigma influence_j * W_ij
  └─ 개선: exposure_score = recSys_rank * social_weight * relevance
```

#### B. 24h Activity Vector

에이전트별 시간대 활성 확률 추가:

```
AgentProfile:
  + activity_vector: list[float]  # 24차원, 각 시간대 활성 확률
```

#### C. 검증 방법론 참조

OASIS가 Twitter15/16으로 검증한 방법론을 Prophet에도 적용:
- Scale (참여 규모), Depth (전파 깊이), Max Breadth (최대 폭) 메트릭으로 실데이터 비교

## 7. 최종 요약

|  | OASIS 채택 | Prophet 자체 | **하이브리드 (추천)** |
|---|---|---|---|
| 개발 속도 | 빠름 | 느림 | 중간 |
| LLM 비용 | 매우 높음 | 낮음 | **낮음** |
| 마케팅 특화 | 낮음 | 높음 | **높음** |
| 확산 예측력 | 낮음 (창발적) | 높음 (수학적) | **높음** |
| 행동 현실성 | 높음 | 중간 | **중상** |
| SaaS 적합성 | 낮음 | 높음 | **높음** |
| 외부 의존도 | 높음 | 없음 | **없음** |
| 확장 자유도 | 제한적 | 완전 | **완전** |

### 결론

> **Prophet의 자체 6-Layer 엔진 + Viral Cascade Algorithm을 유지하되, OASIS의 RecSys 개념과 시간 모델을 차용하는 하이브리드 전략이 최적이다.**
>
> 마케팅 SaaS라는 Prophet의 포지셔닝에서 비용 제어(3-Tier), 수학적 예측(Cascade Detection), 마케팅 메트릭이 핵심 경쟁력이며, 이는 OASIS에는 없는 Prophet 고유의 강점이다.
